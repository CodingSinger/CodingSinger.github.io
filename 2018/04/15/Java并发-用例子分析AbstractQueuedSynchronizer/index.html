Java并发-用例子分析AbstractQueuedSynchronizer

发表于 2018-04-15 | 分类于 java

本文基于jdk1.8，AQS类中两千多行代码，我们不需要理解每一行代码的作用，所以我们根据ReentrantLock一个案例来了解AQS是如何
利用await和signal进行线程之间的同步。

代码很简单，就是实现AB线程之间的顺序关系，保证A、B顺序输出。我们根据该例子深入AQS。

首先假设A线程先获得锁，A执行lock操作，调用了默认的NoFairSync的lock操作。

compareAndSetState操作就是利用CAS操作原子性地比较state的数值是否为0，如果为0，则修改为1，修改成功返回true,
如果之前state不为0，则不进行修改返回false。

前几篇文章我们讲过，state变量通常用来表示一个当前的锁状态。在ReentrantLock类中，state变量被用来表示所有者线程重复获取
该锁的次数。继续，如果修改成功了，则设置当前线程为独占线程。

于是乎，我们退出了lock方法。获取到锁，继续A线程工作。

然后此时B线程调用该锁对象的lock方法，我们清楚地知道此时state变量为1了，所以compareAndSetState操作会失败，而执行AQS类中的acquire(1)
方法。

tryAcquire是在ReentrantLock类中重写的，所以我们略过先不看，我们直接看B线程尝试获取锁失败，tryAcquire返回false，从而直接
执行acquireQueued(addWaiter(Node.EXCLUSIVE),arg)。

AQS类中有head和tail引用，构建了维护了个锁队列。

addWaiter方法将当前线程包装成Node对象，放在等待锁队列的末尾。mode参数来控制当前节点的锁模式，主要有共享锁和独占锁，在代码中表示为

node的前继节点。虽然此处并未用到。

ok，现在我们的B线程在A线程获取独占锁的情况下已经进入等待队列了。然后调用了AQS的acquireQueued方法，该方法真正将该节点的线程置入WAIT状态。

于是调用tryAcquire方法，该方法也是在ReentrantLock中重写了，这里我们不去了解，因为A线程拥有着独占锁，这里B去尝试获取锁肯定是
失败的，于是返回false,接着执行shouldParkAfterFailedAcquire方法。
在将shouldParkAfterFailedAcquire方法之前，我们需要先了解一下node节点的waitStatus属性.

如果不指定waitStatus的值，则waitStatus刚开始时都是0.

ok，还是以我们的代码来讲述，B线程的前置节点的waitStatus一直都没变过，所以默认为0，此时的话是不符合park该B线程的条件的，于是修改前置节点的
waitStatus的值为SIGNAL,所以acquireQueued的第一次循环是不会park该B线程，第二次循环则符合了条件，于是进入parkAndCheckInterrupt。

于是B线程就可怜地被park了，进入WAIT状态。
我们继续来讲A线程，A线程输出完’A’之后调用了ca.await(),我们继续单步。该方法其实调用的是AQS的内部类ConditionObject中的await方法。

我们先来了解下ConditionObject这个内部类，ReentrantLock上调用newCondition其实返回的就是这个类的实例对象。
我们说过AQS上维护了head和tail引用来抽象一个等待队列。
ConditionObject类中维护了firstWaiter和lastWaiter引用来构造一个等待在该ConditionObject实例上的等待唤醒队列(就是在该实例上调用了await方法的线程)

addConditionWaiter方法将当前线程包装成Node实例并加入到调用的ConditionObject的等待唤醒队列中。

于是B线程自己进入了WAIT状态，而又将A线程从等待锁队列中唤醒。

A线程从//1处继续执行，可以看到熟悉的acquireQueued函数，我们也不再分析，该函数在这里主要是重置等待锁队列的头结点。

ok，循环往复，A线程B线程于是就交叉运行了，输出A,B,A,B….

总结：

总的来说，wait和signal以及，lock，unlock的实现原理是利用了AQS和ConditionObject中的两个队列，一个是由AQS维护的等待锁队列，
另一个则是由ConditionObject维护的该对象实例的等待唤醒队列。然后通过两个队列的元素的进队出队以及用LockSupport的park和unpark方法
实现了AQS的核心功能。当然这之间需要考虑到很多问题，例如线程的waitStatus的状态改变检测以及Thread的中断。
